# 系统架构文档

## 整体架构

本系统采用四步诊断流程，结合可演化专家池（EEP）和知识增强技术，实现多专科医疗诊断辅助。

```
┌─────────────────────────────────────────────────────────────┐
│                      诊断流水线                              │
└─────────────────────────────────────────────────────────────┘
                            │
        ┌───────────────────┼───────────────────┐
        │                   │                   │
        ▼                   ▼                   ▼
┌──────────────┐   ┌──────────────┐   ┌──────────────┐
│ 可演化专家池  │   │  知识检索     │   │  病例库/经验库│
│ (28位专家)   │   │  (RAG)       │   │              │
└──────────────┘   └──────────────┘   └──────────────┘
```

## 核心组件

### 1. 四步诊断流程

#### Step 1: 科室路由 (system_step1_route.py)
- **功能**: 分析患者症状，确定最相关的临床科室
- **输出**: 科室名称及置信度
- **技术**: 基于规则的路由 + LLM语义理解

#### Step 2: 信息重构 (system_step2_ir.py)
- **功能**: 将原始病历重构为结构化的诊断信息
- **输出**: 重构后的患者信息
- **技术**: LLM提示工程 + 信息提取

#### Step 3: 专家诊断 (system_step3_diag.py)
- **功能**: 调用专科专家进行诊断推理
- **输出**: 鉴别诊断、风险评估、检查建议
- **技术**: 可演化专家池 + 知识检索

#### Step 4: 结果聚合 (system_step4_agg.py)
- **功能**: 整合多位专家意见，生成最终诊断
- **输出**: 综合诊断结果
- **技术**: 多专家投票 + 置信度加权

### 2. 可演化专家池 (EEP)

#### 专家池结构
```json
{
  "id": "oncology_1",
  "name": "肿瘤科专家（变体1）",
  "specialty": "肿瘤科",
  "description": "擅长各类良恶性肿瘤的内科综合治疗",
  "focus_areas": ["放化疗方案", "肿瘤筛查", "癌痛管理", "MDT"],
  "thinking_patterns": ["肿瘤标志物解读", "TNM分期", "基因突变分析"],
  "evolved_prompt": "...",  // 优化后的诊断提示词
  "fitness": 0.976,
  "stats": {
    "accuracy": 1.0,
    "avg_score": 92.0,
    "valid_count": 5
  }
}
```

#### 专家激活模式
- **语义激活**: 基于症状描述的语义相似度
- **规则激活**: 基于科室匹配的确定性激活
- **混合激活**: 结合语义和规则

### 3. 遗传算法优化

#### 算法流程
```
初始化种群 (N=16)
  │
  ├─→ 评估适应度 (在验证集上测试)
  │
  ├─→ 选择 (精英保留 + 锦标赛选择)
  │
  ├─→ 交叉 (融合两个优秀Prompt)
  │
  ├─→ 变异 (调整Prompt内容)
  │
  └─→ 重复G代 (G=10)
```

#### 适应度函数
```python
fitness = accuracy * 0.7 + normalized_score * 0.3
```
- `accuracy`: 诊断准确率（0-1）
- `normalized_score`: 归一化诊断质量分数（0-1）

#### 遗传操作
1. **交叉**: 融合两个父代Prompt的优点
2. **变异**: 调整诊断策略、输出格式、推理原则
3. **选择**: 精英保留 + 适应度比例选择

### 4. 知识增强

#### RAG (检索增强生成)
- **索引构建**: FAISS向量索引
- **检索策略**: Top-K相似文档
- **融合方式**: 将检索到的指南注入诊断上下文

#### 经验库
- **存储**: 历史成功诊断案例
- **检索**: 症状相似度匹配
- **应用**: 提供参考案例

#### 病例库
- **格式**: 结构化病例记录
- **检索**: 多字段匹配
- **应用**: 类比推理

## 性能优化

### 1. 并发处理
- 多专家并发诊断（Step 3）
- 批量患者并发处理

### 2. 缓存机制
- Step 2重构结果缓存
- RAG检索结果缓存
- LLM响应缓存

### 3. 早停机制
- 遗传算法连续N代无改进时提前终止
- 减少不必要的计算

## 扩展性设计

### 添加新科室
1. 在 `EXPERT_TEMPLATES` 中定义专家模版
2. 准备该科室的验证病例
3. 运行遗传算法优化
4. 将结果合并到专家池

### 添加新知识源
1. 实现新的检索器接口
2. 在 `KnowledgeRetriever` 中注册
3. 配置激活条件

### 调整诊断流程
1. 修改对应的 `system_step*.py`
2. 更新 `DiagnosticPipeline` 中的调用逻辑
3. 重新测试整体性能

## 技术栈

- **语言**: Python 3.8+
- **LLM接口**: OpenAI API / 兼容接口
- **向量检索**: FAISS
- **数据处理**: Pandas, NumPy
- **并发**: asyncio, concurrent.futures

## 配置参数

主要配置参数位于各模块的类初始化中：

```python
# DiagnosticPipeline配置
pipeline = DiagnosticPipeline(
    activation_mode="eep_semantic",  # 专家激活模式
    enable_rag=True,                 # 是否启用RAG
    enable_experience=False,         # 是否启用经验库
    enable_case=False,               # 是否启用病例库
    top_k_experts=2,                 # 激活专家数量
    llm_model="gpt-4"               # LLM模型
)

# 遗传算法配置
optimizer = GeneticPromptOptimizer(
    base_prompt=initial_prompt,
    pipeline=pipeline,
    population_size=16,              # 种群大小
    elitism_count=2,                 # 精英保留数
    mutation_rate=0.3,               # 变异率
    crossover_rate=0.7               # 交叉率
)
```
